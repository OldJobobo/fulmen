# Fulmen Pathworking Model

Fulmen began as a Tree of Life agent harness. The current runtime flow is mostly
the Lightning Flash: intention descends into a concrete artifact. A fuller
pathworking mode adds the missing ascent: start from what has already manifested,
climb back toward root intention, then descend again with clearer purpose.

## Sephirothic Roles

| Sephirah | Fulmen role |
|---|---|
| Kether | Root objective or recovered true intention |
| Chokmah | Raw possibility and expansive ideation |
| Binah | Structural constraint and form |
| Da'at | Compressed brief: goal, constraints, scope, and definition of done |
| Chesed | Planner: expansive plan, resources, options |
| Geburah | Critic: restriction, pruning, risk, judgment |
| Tiphareth | Parent session: integration and coherent orchestration |
| Netzach | Worker: generative execution |
| Hod | Explorer and Verifier: mapping, analysis, structure, validation |
| Yesod | Assembler: synthesis and transmission into deliverable form |
| Malkuth | Final output, committed action, tool result, or visible artifact |

The verifier is a Hod specialization. Explorer uses Hod before execution to map
structure; verifier uses Hod after execution to test claims, expose unstable
work, and confirm that the manifestation still matches the intended form.

## Descent: Lightning Flash

Execution mode moves from intention to manifestation:

```text
Kether -> Da'at -> Chesed/Geburah -> Tiphareth -> Netzach/Hod -> Yesod -> Malkuth
```

In practical Fulmen terms:

```text
root task -> brief -> planner/critic -> integrated plan -> worker/verifier -> assembler -> final artifact
```

This mode is appropriate when the user's goal is already clear and the main need
is disciplined execution.

## Ascent: Pathworking Diagnosis

Pathworking mode starts from reality instead of assumption:

```text
Malkuth -> Yesod -> Hod/Netzach -> Tiphareth -> Geburah/Chesed -> Da'at -> Kether
```

In practical Fulmen terms:

```text
current artifact -> extracted pattern -> analysis and affordances -> diagnosis -> limits/options -> compressed understanding -> root intention
```

This mode is appropriate when the task is unclear, architectural, diagnostic,
creative, symbolic, or when the existing artifact needs to be understood before
it can be changed.

## Full Pathworking Loop

The deeper Fulmen pattern is a double movement:

```text
Malkuth -> Kether -> Malkuth
```

Expanded:

```text
current reality
  -> pattern extraction
  -> structural analysis and creative affordance
  -> integrated diagnosis
  -> recovered root intention
  -> compressed brief
  -> expansive plan and adversarial constraint
  -> scoped execution and verification
  -> assembled final artifact
```

This makes Fulmen less purely top-down. The harness first asks what has actually
manifested, then recovers what it is trying to become, then descends again into a
more intentional result.

## Path Transforms

Pathworking treats handoffs as transformations, not raw message passing.

Useful Fulmen transforms:

| Path | Transform |
|---|---|
| Chesed -> Geburah | Subject an expansive plan to adversarial critique |
| Geburah -> Tiphareth | Integrate constraints into a coherent working plan |
| Tiphareth -> Netzach | Reduce the integrated plan to a scoped implementation task |
| Tiphareth -> Hod | Reduce the integrated plan to a scoped mapping or verification task |
| Netzach -> Hod | Pass generative output through analytical compression and verification |
| Hod -> Yesod | Structure findings into a consistent assembly format |
| Yesod -> Malkuth | Transmit the assembled result into the final artifact |
| Malkuth -> Yesod | Extract the pattern implied by the current artifact |
| Yesod -> Hod | Analyze the structure, defects, and constraints of that pattern |
| Hod/Netzach -> Tiphareth | Integrate structural truth and creative direction into diagnosis |
| Tiphareth -> Da'at | Compress diagnosis into operational understanding |
| Da'at -> Kether | Recover the root intention behind the work |

## Operating Guidance

Use simple Lightning Flash execution for clear tasks.

Use the full Malkuth -> Kether -> Malkuth loop when Fulmen needs to understand an
existing artifact, resolve ambiguity, redesign an architecture, or preserve a
symbolic/conceptual lineage while making practical changes.
